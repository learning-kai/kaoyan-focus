# 项目开发规范（AI协作）

本文档面向 AI 与开发者，规定在 `考研专注` 仓库中开发、重构、修复、发版和更新文档时必须遵守的规则。

阅读顺序要求：

1. [项目文件结构说明.md](./项目文件结构说明.md)
2. [项目完整链路说明.md](./项目完整链路说明.md)
3. 当前文件
4. 如涉及 PR，再阅读 [开发者AI开发与PR提交流程.md](./开发者AI开发与PR提交流程.md)

原则：

- 目标是让项目更清晰、更可维护、更可测试，不是单纯把代码拆碎。
- 桌面端是当前公开仓库主线。
- 重构优先考虑稳定性、职责边界和可理解性。
- 乱码问题视为阻塞问题，不是后续顺手修的小问题。
- 不提交私钥、真实数据库、日志、构建产物或个人敏感信息。

## 0. AI 协作执行协议

### 0.1 开发前必须实际阅读

任何涉及代码、流程、UI、配置、测试、文档、提交、发布的任务，开始前必须实际阅读或重新核对：

- [项目文件结构说明.md](./项目文件结构说明.md)
- [项目完整链路说明.md](./项目完整链路说明.md)
- 当前文件

不能只凭历史记忆、上一次会话摘要或“看起来知道项目”直接开发。

如果用户明确说“只分析”“先看看”“暂时不改”，只能输出分析，不得擅自改代码。  
如果用户明确要求“实现”“修复”“按计划执行”，应推进到实现、验证和清晰回执。

### 0.2 工作区脏改动规则

- 开始前必须查看 `git status --short`。
- 遇到已有脏改动，默认认为是用户或上一轮任务留下的，不能擅自回滚。
- 如果脏改动与本任务无关，只忽略。
- 如果脏改动影响当前任务，必须先理解并基于它继续工作。
- 不得使用 `git reset --hard` 或 `git checkout --` 回滚用户改动，除非用户明确要求。

### 0.3 阶段化开发要求

复杂任务必须分阶段推进：

1. 读代码和文档，确认真实链路。
2. 列出改动范围和风险点。
3. 按模块实施。
4. 每完成一段做定向检查。
5. 最后跑必要测试并总结。

阶段自检未通过时，不能进入下一阶段。

### 0.4 分析与实现不能脱节

分析中确认的边界，开发时必须落到代码和文档里。

例如：

- 如果确认学习模式状态机在 Rust 后端，就不能把核心计时逻辑塞回前端。
- 如果确认 Tauri command 是通信契约，就不能为了“更好看”改命令名。
- 如果确认某个字段参与同步，就不能只改本地表而漏掉 payload。

## 1. 架构原则

### 1.1 前端原则

- 页面负责交互和展示，核心业务事实来自 Rust 和 SQLite。
- `services/` 是前端调用后端的边界，页面不直接拼 Tauri command 字符串。
- 设置页只负责组合和状态协调，领域设置放在 `pages/settings/` 面板中。
- 浏览器预览只能用于 UI 检查，不代表桌面能力可用。

### 1.2 Rust 后端原则

- `lib.rs` 只做 Tauri 装配、插件、托盘、窗口事件和命令注册。
- 后台循环统一放在 `background_tasks.rs`。
- Windows 原生能力集中在 `windows/`。
- 数据库 schema 和兼容迁移集中在 `storage/db.rs`。
- Tauri command 按业务域放在 `commands/`。

### 1.3 数据库原则

- SQLite 是本地事实来源。
- 表结构变更必须兼容已有用户数据库。
- 不允许直接删除用户数据字段。
- 新增列必须使用兼容迁移。
- 涉及同步的实体必须同时考虑 `sync_meta`、导出、导入、合并和软删除。

### 1.4 同步原则

- 不随意改变同步 payload 形状。
- 不随意改变 Tauri command 入参和返回结构。
- active study merge 是高风险区域，任何相关修改必须跑 Rust 同步测试。
- 备份恢复必须先校验，再替换或导入。

### 1.5 桌面能力边界

项目是用户自愿使用的自律工具，不是系统管控软件。

禁止实现：

- 隐藏进程。
- 驱动级拦截。
- 阻止任务管理器结束进程。
- 阻止系统关机、注销、重启。
- 恶意持久化。

## 2. 模块边界规则

### 2.1 可以增长但要保持清晰的文件

- `src/styles.css`：全局样式可以集中，但新增样式必须有明确页面或组件归属。
- `src/types/settings.ts`：设置相关类型可以暂时集中，但新增类型要保持命名清晰。
- `src-tauri/src/storage/db.rs`：schema 和迁移可以集中，但必须按表和迁移目的分段。

### 2.2 不应继续膨胀的区域

- `SettingsPage.tsx` 不应重新堆回所有设置 UI。
- `settingsApi.ts` 不应重新混入同步、飞书、邮件和健康接口。
- `lib.rs` 不应重新塞后台循环。
- `sync_package/identity.rs` 和 `commands/sync/object_storage_protocol.rs` 是下一轮深拆候选。

### 2.3 新增文件规则

新增文件后必须检查：

- 是否放在正确领域目录。
- 是否需要从 `mod.rs` 或 index 文件导出。
- 是否需要更新 [项目文件结构说明.md](./项目文件结构说明.md)。
- 是否需要补测试或文档。

## 3. 新增功能接入规范

### 3.1 新增页面功能

新增页面功能时，至少检查：

- 页面组件。
- service API。
- Rust command。
- `lib.rs` command 注册。
- 前端类型。
- 浏览器预览兜底。
- 学习模式运行中是否需要锁定。

### 3.2 新增设置项

新增设置项时，至少检查：

- 默认值。
- 前端类型。
- 设置页 UI。
- settings 或领域 API。
- Rust 读写逻辑。
- SQLite `settings` 兼容。
- 是否参与同步。
- 是否需要隐藏敏感值。

### 3.3 新增同步实体

新增同步实体时，至少检查：

- SQLite 表和软删除字段。
- `sync_meta`。
- `SharedSyncPayload`。
- export/import/merge。
- 对象存储操作日志。
- 冲突策略。
- 测试。

### 3.4 新增第三方集成

新增第三方集成时，必须明确：

- 凭据保存方式。
- 设置页展示方式。
- 手动测试入口。
- 自动任务是否需要后台调度。
- 错误是否进入运行健康。
- 日志中是否会泄露 token、secret、邮箱密码或远端 ID。

## 4. 测试规范

### 4.1 默认检查

常规改动完成后优先运行：

```powershell
npm.cmd test
npm.cmd run build
cd src-tauri
cargo test
```

### 4.2 定向检查

- 前端 UI 改动：至少做浏览器预览或 Tauri 桌面 smoke。
- 学习模式改动：检查启动、暂停、恢复、确认休息、完成、重启恢复。
- 白名单改动：检查软件、网站、PotPlayer、目录和大小写匹配。
- 同步改动：检查手动上传、手动下载、自动同步、备份预检、恢复。
- 飞书改动：检查设置保存、OAuth、手动同步、重建、运行记录。
- 发布改动：跑 release dry run。

### 4.3 不允许的做法

- 测试失败后只说“可能是环境问题”而不说明具体失败点。
- 没跑测试却写“测试通过”。
- 因为是文档改动就忽略 package 元数据或链接检查。
- 用真实密钥、真实数据库或私人账号做公开测试材料。

## 5. 文档更新规范

必须更新文档的场景：

- 新增、删除、重命名核心文件。
- 改业务链路。
- 改同步协议或 payload。
- 改数据库 schema。
- 改发布流程。
- 改用户可见功能。
- 改贡献流程或安全边界。

对应关系：

- 文件结构变化：更新 [项目文件结构说明.md](./项目文件结构说明.md)。
- 链路变化：更新 [项目完整链路说明.md](./项目完整链路说明.md)。
- 开发规则变化：更新当前文件。
- PR 流程变化：更新 [开发者AI开发与PR提交流程.md](./开发者AI开发与PR提交流程.md)。
- 用户功能变化：更新 [FEATURES.md](./FEATURES.md)。

## 6. 命名规范

- Tauri command 使用现有 snake_case 命名，不随意改名。
- 前端 service 函数使用清晰动词，如 `get/save/test/list/sync/restore`。
- 设置 key 使用稳定 snake_case。
- 同步 entity type 必须稳定，不因 UI 文案变化而改名。
- 文件命名按领域和职责，不使用“new”“temp”“final”一类临时名。

## 7. 安全与公开仓库要求

公开仓库不得提交：

- `.env*`
- `*.log`
- `dist/`
- `node_modules/`
- `src-tauri/target/`
- `src-tauri/updater/tauri-update.key`
- 真实 SQLite 数据库
- WebDAV、对象存储、飞书、SMTP 凭据
- 包含个人信息的截图

公开仓库可以提交：

- `src-tauri/updater/tauri-update.key.pub`
- 脱敏截图
- 示例配置说明

## 8. AI 自检清单

最终回复前必须确认：

- 是否只改了用户要求范围。
- 是否误碰无关脏改动。
- 是否更新了相关文档。
- 是否保留了 Tauri command 名称。
- 是否保留了 SQLite 兼容性。
- 是否无乱码。
- 是否运行了必要测试。
- 如果未运行测试，必须明确说明原因。

## 9. 完成标准

一次任务完成时，最终回执至少说明：

- 做了什么。
- 关键文件。
- 验证命令和结果。
- 未完成或仍有风险的点。

不要用“应该可以”“大概没问题”替代真实检查结果。
