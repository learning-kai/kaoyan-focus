# 项目完整链路说明

本文档面向 AI 与开发者，目标是让阅读者在最短时间内理解 `考研专注` 做什么、怎么跑、数据怎么流、哪些边界不能误改。

建议阅读顺序：

1. [项目文件结构说明.md](./项目文件结构说明.md)
2. 当前文件
3. [项目开发规范（AI协作）.md](./项目开发规范（AI协作）.md)

## 1. 项目目标

`考研专注` 是一个 Windows 桌面端本地学习约束工具。它的核心价值不是单个番茄钟，而是把学习准备、前台规则强制执行、计划任务、复盘统计和可选云同步串成一个闭环。

核心链路：

- 用户配置学习节奏、科目和前台规则。
- 用户启动学习模式。
- Rust 后端持久化学习模式状态，并由后台 tick 推进阶段。
- 前台应用检测持续判断当前窗口是否符合白名单或黑名单模式。
- 清单、课表、复盘、统计围绕学习记录形成回看。
- WebDAV、对象存储、飞书、邮件提醒作为可选集成接入。

明确边界：

- 不隐藏进程。
- 不绕过 Windows 权限。
- 不阻止任务管理器结束进程。
- 不阻止系统关机、注销、重启。
- 不做驱动级拦截。
- 不上传用户隐私数据，除非用户主动配置同步或第三方集成。

## 2. 核心运行参与者

### 2.1 React 前端

前端负责页面展示、用户输入、状态轮询、Tauri 命令调用和浏览器预览兜底。

主要参与者：

- `App.tsx`：页面装配、全局自动任务调度、提醒、自动同步、飞书刷新。
- `pages/FocusPage.tsx`：学习模式主入口。
- `pages/SettingsPage.tsx` 和 `pages/settings/`：设置页组合和分区面板。
- `services/`：前端到 Tauri command 的边界。

原则：

- 前端不直接维护学习模式核心时间状态。
- 前端可以展示倒计时，但权威状态来自 Rust 和 SQLite。
- 浏览器预览只能检查 UI，不能代表桌面能力可用。

### 2.2 Rust 后端

Rust 后端负责本地数据库、Windows 能力、学习模式状态机、同步协议、飞书和邮件等桌面能力。

主要参与者：

- `lib.rs`：Tauri 装配和命令注册。
- `background_tasks.rs`：后台 tick、对象存储轮询、备份清理。
- `storage/db.rs`：SQLite schema 和兼容迁移。
- `commands/*`：Tauri command 实现。
- `windows/*`：Windows 原生能力封装。

原则：

- Tauri command 名称是前后端通信契约，不能随意改名。
- SQLite 表结构变更必须兼容已有用户数据库。
- 学习模式状态机不能散落到前端。

### 2.3 SQLite 本地数据库

数据库是应用的本地事实来源。

核心表：

- `focus_sessions`：专注记录。
- `study_modes`：学习模式状态机记录。
- `subjects`：科目。
- `checklist_tasks`、`today_plan_items`：清单和今日任务。
- `schedule_templates`、`schedule_blocks`：课表。
- `daily_reviews`、`weekly_reviews`：复盘。
- `whitelist_apps`：软件、网站、PotPlayer 前台规则；白名单/黑名单模式共用同一张表。
- `app_events`：干扰和应用事件。
- `settings`：应用设置和集成设置。
- `sync_meta`、`sync_runs`、`sync_applied_ops`、`sync_entity_versions`：同步元数据。
- `feishu_sync_links`、`feishu_sync_runs`：飞书同步链接和运行记录。

## 3. 启动链路

应用启动后，Tauri builder 完成插件、托盘、窗口事件和命令注册。

setup 阶段会启动后台任务：

- 学习模式 tick：定期推进学习模式阶段。
- 对象存储轮询：检查远端是否有新同步数据。
- 同步备份清理：清理过期本地和远端备份。

前端启动后：

- 初始化主题。
- 加载页面导航。
- 周期性检查闹钟、提醒、自动同步、飞书同步和自动更新。
- 如果处于浏览器预览环境，系统能力会显示“桌面壳未连接”。

## 4. 学习模式链路

学习模式是项目最重要的状态机。

启动流程：

1. 前端在专注页读取默认设置、科目和当前学习模式状态。
2. 用户启动学习模式。
3. 前端调用 focus 相关 Tauri command。
4. Rust 校验当前设备是否允许启动。
5. Rust 创建 `study_modes` 记录和当前 `focus_sessions` 记录。
6. 后端运行态记录当前学习模式 active 和 session id。
7. 前端展示极简运行态界面。
8. 后台 tick 根据时间推进 `focus`、`awaiting_break`、`break`、`finished`。

阶段含义：

- `idle`：未运行。
- `focus`：专注中。
- `awaiting_break`：本轮专注到点，等待用户确认休息。
- `break`：休息中。
- `finished`：学习模式完成。
- `emergency_exited`：应急退出状态。

维护要求：

- 学习模式阶段推进必须以数据库和 Rust 后端为准。
- 运行中设置页和前台规则页只能查看，不能改配置。
- 主设备控制、远端同步控制、暂停恢复等逻辑必须保留状态版本和时间单调性。

## 5. 前台规则与检测链路

前台规则链路负责判断学习期间当前前台应用是否允许使用。规则条目复用 `whitelist_apps`，全局设置 `whitelist_mode` 决定语义：`allowlist` 表示只放行命中规则，`blocklist` 表示只拦截命中规则。

流程：

1. 后台或前端触发前台检测。
2. Rust 通过 Windows API 获取前台窗口、进程、标题和可选浏览器 URL。
3. 匹配软件规则、网站规则、PotPlayer 视频规则。
4. 根据 `allowlist` 或 `blocklist` 语义得到允许/拦截结果。
5. 如果被拦截，按当前学习模式约束记录干扰或提醒。

维护要求：

- Windows 能力应集中在 `windows/`，不要散落在页面命令里。
- 前台规则匹配应优先补测试，尤其是路径、大小写、目录和 PotPlayer 规则。
- 不能实现恶意级系统锁死能力。

## 6. 清单、课表与复盘链路

清单、课表和复盘是学习数据的输入与回看层。

清单链路：

- 清单页面维护五类待办和今日任务。
- 本地改动写入 SQLite 后触发对象存储同步和飞书同步。
- 今日任务可以被飞书任务映射。

课表链路：

- 课表模板负责重复计划。
- 课表块负责具体日期安排。
- 课表块可以映射到飞书日历事件。

复盘链路：

- 每日复盘记录当天总结、明日重点等内容。
- 周复盘记录一周视角。
- 复盘数据参与同步包导入导出。

维护要求：

- 新增字段要同步考虑本地 SQLite、同步 payload、飞书映射和前端类型。
- 删除或软删除语义必须和 `sync_meta` 保持一致。

## 7. 同步链路

同步分为 WebDAV 和对象存储两条后端。

### 7.1 WebDAV

WebDAV 主要同步数据库文件。

流程：

1. 设置页保存 WebDAV 地址、用户名、密码和远端路径。
2. 手动上传时读取本地数据库文件并 PUT 到远端。
3. 手动下载时先备份本地数据库，再下载远端文件并校验 SQLite。
4. 自动同步根据远端时间戳和本地修改时间判断上传或下载。

### 7.2 对象存储 R2/S3

对象存储使用共享 payload、操作日志、manifest、snapshot 和 active lock。

流程：

1. 从 SQLite 导出 `SharedSyncPayload`。
2. 将本地实体转换为操作记录。
3. 读取远端 manifest、snapshot 和操作日志。
4. 合并远端操作和本地 payload。
5. 导入合并结果到 SQLite。
6. 写入新操作、snapshot、manifest 和同步运行记录。
7. 必要时写入备份，支持预检和恢复。

维护要求：

- 不改变同步 payload 形状，除非同时提供兼容策略。
- active study merge 是高风险区域，必须跑 `cargo test` 中的同步测试。
- 备份恢复必须先校验，再替换或导入。

## 8. 飞书链路

飞书是可选集成，用于同步本地任务和课表到飞书任务清单与日历。

流程：

1. 设置页保存飞书 App ID、Secret、Redirect URI 和启用状态。
2. 用户启动浏览器授权。
3. 本地 OAuth callback 获取 code。
4. 后端交换 token 并保存。
5. 同步时确保飞书任务清单和日历容器存在。
6. 本地清单任务映射为飞书任务。
7. 本地课表块映射为飞书日历事件。
8. link 表记录本地实体、远端 ID、父容器、远端最后修改时间和内容指纹。
9. 同步结果写入 `feishu_sync_runs`。

当前任务同步的冲突策略：

- 本地任务和远端任务都计算内容指纹，字段覆盖标题、备注、截止日期、完成状态和任务清单。
- `feishu_sync_links.remote_etag` 保存上次同步后的本地内容基线，`remote_change_key` 保存上次同步后的远端内容基线。
- 同步时先判断本地是否相对基线变化、远端是否相对基线变化；只有一侧变化时直接采用变化侧。
- 双方都变化时再比较时间戳；时间接近且同步由本地变更触发时优先推本地，避免飞书侧 `updated_at` 新但内容未改时覆盖本地修改。
- 日历同步继续沿用已有 fingerprint 基线，任务同步不要退回到单纯比较 `updated_at`。

实时同步方案边界：

- 飞书官方支持事件订阅、长连接接收事件、Webhook 回调和任务更新事件 `task.task.update_user_access_v2`。
- 桌面端通常没有公网回调地址，不能把 Webhook 当成唯一实时方案。
- 合理架构是“本地变更立即推送 + 能接事件时用长连接/Webhook 触发远端拉取 + 接不了事件时短周期轮询兜底”。

维护要求：

- token、secret、远端 ID 不应写入公开文档或截图。
- 飞书删除事件、空标题、孤儿 link 是重点回归场景。
- 任务同步冲突判断必须基于内容指纹基线，不能只看远端更新时间戳。
- 重建飞书任务清单前必须创建本地备份和远端任务清单备份。

## 9. 邮件提醒链路

邮件提醒用于到期任务通知。

流程：

1. 设置页保存 SMTP host、port、安全方式、账号、密码、发件人和收件人。
2. 用户可以发送测试邮件。
3. 前端定期调用到期任务检查。
4. 后端查询需要提醒的任务，发送邮件并记录通知日志。

维护要求：

- SMTP 密码不能出现在日志和文档中。
- 邮件失败应反馈给设置页，但不能影响学习模式主链路。

## 10. 更新发布链路

Tauri updater 使用公钥校验更新资产。

默认更新源：

- Release 仓库：`learning-kai/kaoyan-focus`。
- 下载目录：`https://github.com/learning-kai/kaoyan-focus/releases/latest/download`。
- 自动更新元数据：`https://github.com/learning-kai/kaoyan-focus/releases/latest/download/latest.json`。

发布脚本当前能力：

- 更新版本号。
- 生成 changelog。
- 构建 Windows NSIS 安装包。
- 签名更新包。
- 生成 `latest.json`。
- 上传 GitHub Release。

维护提醒：

- 私钥 `src-tauri/updater/tauri-update.key` 不得提交。
- 公钥 `tauri-update.key.pub` 可以提交。
- 默认发布链路只处理 Windows/Tauri 桌面端。
- Android 版本同步是显式 opt-in：传入 `--include-android` 或设置 `INCLUDE_ANDROID_RELEASE=1` 后才检查 `UltraFocus-master`、同步版本号和校验 tag。
- 发布流程改动必须先跑 `npm.cmd run release:auto -- --version 1.7.4 --dry-run`，确认 `Update base URL` 和 `GitHub update repo` 指向 `learning-kai/kaoyan-focus`。

## 11. 新增功能最容易漏掉的地方

新增设置项时，通常需要同时检查：

- 前端类型。
- 设置页默认值。
- `settingsApi` 或对应领域 API。
- Rust `commands/settings.rs` 或领域命令。
- SQLite `settings` 默认值和迁移。
- 同步 payload 是否需要包含。
- 文档是否需要更新。

新增同步实体时，通常需要同时检查：

- SQLite 表和软删除语义。
- `sync_meta`。
- `SharedSyncPayload`。
- export/import/merge。
- 冲突策略。
- 测试。

新增页面能力时，通常需要同时检查：

- 页面组件。
- service API。
- Rust command。
- `lib.rs` command 注册。
- 浏览器预览兜底。
- 学习模式运行中是否需要锁定。
