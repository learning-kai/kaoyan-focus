# 考研专注 Windows 端软件技术路线

## 1. 项目目标

本项目是一个运行在 Windows 端的考研专注软件，帮助用户在备考期间减少娱乐软件、社交软件和无关网页干扰，形成稳定、可复盘的学习节奏。

核心目标：

- 用户主动开启专注后，在设定时间内进入受约束的学习环境。
- 专注期间只允许使用白名单软件。
- 专注期间不能通过普通关闭按钮、托盘菜单等常规入口退出。
- 专注过程可记录、可统计、可复盘。
- 所有能力限定在用户自愿安装、自愿开启、本机运行的自律工具范围内。

明确边界：

- 不隐藏进程。
- 不绕过 Windows 权限。
- 不阻止任务管理器结束进程。
- 不阻止系统关机、注销、重启。
- 不做驱动级拦截。
- 不做恶意持久化。
- 不上传用户隐私数据。

## 2. 面向 AI 实现的总体原则

为了方便 AI 分阶段稳定实现，本项目采用“小步可运行、每阶段可验收”的路线。

每个阶段必须满足：

1. 目标明确，只做当前阶段需要的功能。
2. 交付物明确，包括新增文件、页面、接口或数据表。
3. 验收标准明确，可以通过人工操作或命令检查确认。
4. 不提前实现后续阶段功能，只预留必要结构。
5. 每阶段完成后保持项目可启动、可编译、无明显死代码。

AI 实现时应优先遵守：

- 先搭骨架，再做核心闭环，再做增强能力。
- 前端页面和后端能力分层实现。
- Windows 原生能力集中封装，避免散落在业务代码中。
- 对“无法退出”采用用户态防退出，不实现恶意级锁死。
- 每阶段都写清楚验证步骤。

## 3. 推荐技术栈

### 3.1 桌面框架

推荐使用：

- Tauri 2
- React
- TypeScript
- Vite
- Rust

理由：

- Tauri 包体积小，适合个人效率工具。
- Rust 后端适合调用 Windows API。
- React + TypeScript 适合快速实现设置页、倒计时页和统计页。
- 前后端边界清晰，适合 AI 分模块开发。

### 3.2 推荐依赖

前端：

- React Router：页面路由。
- Zustand：轻量状态管理。
- Ant Design 或 shadcn/ui 风格组件：快速构建界面。
- dayjs：时间展示。

后端：

- serde：前后端数据序列化。
- rusqlite 或 tauri-plugin-sql：SQLite 数据存储。
- windows crate：调用 Windows API。
- time 或 chrono：时间处理。

第一版建议优先少依赖，能不用复杂状态库就先不用。

## 5. 专注模式等级设计

### 5.1 普通模式

适合初版 MVP。

行为：

- 关闭主窗口时最小化到托盘。
- 专注期间托盘菜单不显示“退出”，或显示但不可点击。
- 检测到非白名单应用时弹出提醒并拉起主窗口。
- 不主动关闭其他程序。

### 5.2 严格模式

适合第二阶段实现。

行为：

- 专注期间隐藏或禁用所有普通退出入口。
- 专注期间不允许手动停止。
- 非白名单应用连续出现时可执行更强提醒。
- 可配置是否最小化非白名单窗口。

限制：

- 不阻止任务管理器。
- 不阻止系统关机。
- 不做进程隐藏。
- 不做权限绕过。

### 5.3 应急退出

严格模式下仍应保留应急机制，避免误设长时间专注后完全无法恢复。

可选方案：

- 输入随机长验证码。
- 等待 60 秒冷静期。
- 记录一次“应急退出”。
- 结束当前专注并标记为失败。

第一版 MVP 可以不实现应急退出，但路线中必须预留状态字段。

## 6. 白名单设计

### 6.1 白名单识别方式

第一版优先使用进程名识别，例如：

- `Word.exe`
- `SumatraPDF.exe`
- `Anki.exe`
- `Code.exe`

后续增强：

- 按完整路径识别。
- 按签名或发布者识别。
- 按窗口标题规则识别。

### 6.2 白名单添加方式

MVP：

- 用户手动输入软件名称和进程名。

第二阶段：

- 从当前运行进程列表中选择。
- 通过文件选择器选择 `.exe`。
- 从最近被拦截应用中一键加入。

### 6.3 默认放行策略

应内置一组系统基础进程放行规则，避免误拦截系统桌面和壳程序。

示例：

- `explorer.exe`
- 本软件自身进程
- 必要的系统输入法或辅助进程

这些默认放行项不应直接显示为普通用户白名单，也不应被用户随意删除。

## 7. Windows 能力实现路线

### 7.1 前台窗口检测

Rust 后端封装 Windows API：

1. 获取当前前台窗口句柄。
2. 通过窗口句柄获取进程 ID。
3. 根据进程 ID 获取进程名。
4. 返回进程名、窗口标题、进程路径。

建议封装路径：

```text
src-tauri/src/windows/foreground.rs
```

前端不直接接触 Windows API，只通过 Tauri command 获取结果。

### 7.2 非白名单处理

MVP：

- 记录一次干扰事件。
- 弹出提醒。
- 将主窗口置前。

第二阶段可选：

- 最小化非白名单窗口。
- 关闭普通非白名单窗口。

默认不强杀进程。关闭应用必须由用户在设置中主动开启。

### 7.3 防退出

Tauri 层处理窗口关闭事件：

- 非专注状态：允许正常退出。
- 专注状态：阻止关闭，改为隐藏窗口或最小化到托盘。

托盘菜单：

- 非专注状态：显示“显示主窗口”“退出”。
- 专注状态：显示“显示主窗口”，不显示或禁用“退出”。

## 8. 时间与恢复设计

### 8.1 时间计算

专注倒计时应以实际经过时长为准。

建议：

- 前端只负责展示剩余时间。
- 后端保存专注开始时间、计划时长和状态。
- 运行中使用单调时钟或后端计时逻辑计算已用时间。
- 系统时间只用于展示和统计归档。

### 8.2 防改时间原则

第一版不做强防作弊，但应避免完全依赖用户可修改的系统时间。

路线：

- MVP：记录开始时间和计划时长，前端每秒刷新展示。
- 第二阶段：引入运行时单调计时，减少改系统时间的影响。
- 第三阶段：如需更强约束，可增加本地异常检测，但不联网校时。

### 8.3 崩溃和重启恢复

软件启动时检查是否存在 `running` 状态的专注会话。

处理规则：

- 如果计划时间已过：标记为 `interrupted` 或按规则结算。
- 如果计划时间未过：恢复专注界面和剩余时间。
- 如果无法判断：提示用户选择恢复或标记异常中断。

第二阶段必须实现该能力，避免专注状态因重启丢失。

### 8.4 睡眠和唤醒

后续应监听系统睡眠/唤醒事件。

建议规则：

- 睡眠时间不计入有效专注时长。
- 唤醒后恢复专注状态。
- 记录一次系统睡眠事件。

## 9. 网站管控路线

浏览器是学习和娱乐共用工具，不能只按 `chrome.exe` 或 `msedge.exe` 判断。

### 9.1 MVP 策略

MVP 只做软件白名单。

如果浏览器在白名单中，默认允许浏览器运行，不检查具体网页。

### 9.2 第二阶段策略

可做浏览器标题关键词提醒：

- 读取前台窗口标题。
- 根据标题关键字做简单提醒。
- 适合识别部分网页标题，但准确率有限。

### 9.3 第三阶段策略

推荐开发浏览器插件：

- Chrome / Edge 插件读取当前标签页 URL。
- 插件与本地应用通信。
- 本地应用维护网站白名单。
- 非白名单网站显示拦截页或提醒页。

不建议第一版就做本地代理或 HTTPS 拦截，复杂度高且容易引入证书和隐私问题。

## 10. 数据库设计

### 10.1 whitelist_apps

```sql
CREATE TABLE whitelist_apps (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL,
  process_name TEXT NOT NULL,
  path TEXT,
  match_type TEXT NOT NULL DEFAULT 'process_name',
  note TEXT,
  enabled INTEGER NOT NULL DEFAULT 1,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

### 10.2 focus_sessions

```sql
CREATE TABLE focus_sessions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  mode TEXT NOT NULL DEFAULT 'normal',
  subject_id INTEGER,
  planned_seconds INTEGER NOT NULL,
  actual_seconds INTEGER NOT NULL DEFAULT 0,
  started_at TEXT NOT NULL,
  ended_at TEXT,
  status TEXT NOT NULL,
  end_reason TEXT,
  interruption_count INTEGER NOT NULL DEFAULT 0,
  emergency_exit_count INTEGER NOT NULL DEFAULT 0,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

状态建议：

- `idle`
- `running`
- `finished`
- `interrupted`
- `emergency_exited`

### 10.3 app_events

```sql
CREATE TABLE app_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id INTEGER NOT NULL,
  process_name TEXT NOT NULL,
  process_path TEXT,
  window_title TEXT,
  event_type TEXT NOT NULL,
  action_taken TEXT,
  created_at TEXT NOT NULL,
  FOREIGN KEY (session_id) REFERENCES focus_sessions(id)
);
```

CREATE TABLE subjects (

```

### 10.5 settings

CREATE TABLE settings (

```

## 12. AI 分阶段实施路线

### 阶段 0：仓库与文档

目标：建立项目起点。

交付物：

- Git 仓库。
- `技术路线.md`。

验收标准：

- `git status` 能正常工作。
- 文档包含产品目标、技术栈、阶段路线和安全边界。

### 阶段 1：Tauri + React 项目骨架

目标：创建可启动的桌面应用空壳。

实现内容：

- 初始化 Tauri 2 + React + TypeScript + Vite。
- 创建基础页面布局。
- 创建四个页面：专注、白名单、统计、设置。
- 配置基础路由。
- 添加一个 Rust command，用于前端调用测试。

交付物：

- `package.json`
- `src/`
- `src-tauri/`
- 基础页面文件

验收标准：

- 应用能启动。
- 页面能切换。
- 前端能成功调用 Rust command。
- 无 TypeScript 编译错误。

### 阶段 2：专注倒计时闭环

目标：实现不依赖 Windows API 的专注计时 MVP。

实现内容：

- 设置专注时长。
- 开始专注。
- 展示倒计时。
- 专注完成后显示完成状态。
- 保存专注记录到 SQLite。

交付物：

- `FocusPage.tsx`
- `focusApi.ts`
- `src-tauri/src/commands/focus.rs`
- `src-tauri/src/focus/`
- `src-tauri/src/storage/`

验收标准：

- 可以完成一次 1 分钟专注。
- 完成后数据库产生一条 `focus_sessions` 记录。
- 刷新或重启应用后历史记录仍存在。

### 阶段 3：白名单管理

目标：实现软件白名单的增删改查。

实现内容：

- 白名单列表展示。
- 添加软件名称和进程名。
- 启用、停用、删除白名单项。
- 后端持久化到 SQLite。

交付物：

- `WhitelistPage.tsx`
- `whitelistApi.ts`
- `src-tauri/src/commands/whitelist.rs`
- `src-tauri/src/whitelist/`

验收标准：

- 可以新增、删除、启用、停用白名单软件。
- 重启应用后白名单仍存在。
- 空白名单状态有明确提示。

### 阶段 4：Windows 前台应用检测

目标：获取当前前台窗口所属进程。

实现内容：

- Rust 封装 Windows API。
- 返回当前前台窗口标题、进程名、进程路径。
- 前端设置页或调试区域显示当前前台应用。

交付物：

- `src-tauri/src/windows/foreground.rs`
- `src-tauri/src/windows/process.rs`
- `src-tauri/src/commands/monitor.rs`

验收标准：

- 切换到不同应用后，软件能识别前台进程名。
- 对常见应用如浏览器、记事本、PDF 阅读器识别正常。
- 普通用户权限下可运行。

### 阶段 5：专注期间白名单监控

目标：将专注计时和前台检测串起来。

实现内容：

- 专注中每 1 到 3 秒检测一次前台应用。
- 如果应用不在白名单或默认放行列表中，记录干扰事件。
- 前端显示提醒。
- 主窗口拉回前台。

交付物：

- `src-tauri/src/monitor/foreground_monitor.rs`
- `src-tauri/src/whitelist/matcher.rs`
- `app_events` 写入逻辑

验收标准：

- 白名单应用不会触发提醒。
- 非白名单应用会触发提醒。
- 干扰次数会计入当前专注记录。
- CPU 占用保持较低，不因监控明显升高。

### 阶段 6：防普通退出与托盘

目标：实现专注期间无法通过普通入口退出。

实现内容：

- 添加系统托盘。
- 关闭窗口时根据专注状态决定退出或隐藏。
- 专注期间禁用托盘退出。
- 专注结束后恢复正常退出。

交付物：

- `src-tauri/src/tray/menu.rs`
- Tauri 窗口关闭事件处理
- 专注状态与托盘状态联动

验收标准：

- 非专注状态可以正常退出。
- 专注状态点击关闭不会退出。
- 专注状态托盘不能退出。
- 专注结束后可以退出。

### 阶段 7：统计与科目

目标：让软件具备考研属性，而不是普通计时器。

实现内容：

- 添加科目：政治、英语、数学、专业课。
- 开始专注时选择科目。
- 统计今日、本周、本月学习时长。
- 按科目展示累计时长。

交付物：

- `StatsPage.tsx`
- `statsApi.ts`
- `subjects` 表
- `src-tauri/src/commands/stats.rs`

验收标准：

- 可以按科目记录专注。
- 今日、本周、本月统计正确。
- 重启应用后统计不丢失。

### 阶段 8：严格模式、恢复和应急退出

目标：提高专注约束能力和异常恢复能力。

实现内容：

- 增加普通模式/严格模式选择。
- 严格模式下不允许普通停止。
- 实现应急退出流程。
- 启动时恢复未完成专注会话。
- 记录应急退出和异常中断。

交付物：

- 专注模式设置。
- 应急退出 UI。
- 启动恢复逻辑。
- `focus_sessions` 状态完善。

验收标准：

- 严格模式下无法通过普通按钮停止专注。
- 应急退出会留下记录。
- 软件重启后能识别未完成专注。
- 已过期专注会话能正确结算或标记。

### 阶段 9：打包发布

目标：形成可安装的 Windows 软件。

实现内容：

- 配置 Tauri Windows 打包。
- 确定用户数据目录。
- 设置应用图标和名称。
- 测试安装、启动、卸载。
- 确认卸载时是否保留用户数据。

交付物：

- Windows 安装包。
- 发布说明。
- 基础使用说明。

验收标准：

- 全新 Windows 用户环境可以安装启动。
- 卸载流程正常。
- 用户数据保存路径明确。
- 没有管理员权限依赖。

### 阶段 10：网站白名单

目标：解决浏览器既能学习也能娱乐的问题。

实现内容：

- 设计 Chrome / Edge 浏览器插件。
- 插件读取当前标签页 URL。
- 插件与本地应用通信。
- 本地应用维护网站白名单。
- 非白名单网站显示提醒或拦截页。

交付物：

- 浏览器插件项目。
- 本地通信接口。
- 网站白名单页面。

验收标准：

- 浏览器访问白名单网站不提醒。
- 访问非白名单网站能提醒或拦截。
- 插件未安装时主程序仍可正常使用。

## 13. 全局验收标准

MVP 完成标准：

- 能启动 Windows 桌面应用。
- 能创建并完成一次专注。
- 能配置软件白名单。
- 专注期间能检测非白名单前台应用。
- 专注期间无法通过普通关闭或托盘退出。
- 能保存专注记录和基础统计。

成熟版完成标准：

- 有严格模式。
- 有应急退出。
- 有崩溃/重启恢复。
- 有科目统计。
- 有 Windows 安装包。
- 有明确的数据保存位置。
- 有网站白名单路线或插件实现。

## 14. 测试策略

### 14.1 手工端到端测试

每个版本至少测试：

1. 启动应用。
2. 添加白名单。
3. 开始 1 分钟专注。
4. 切换到白名单应用。
5. 切换到非白名单应用。
6. 尝试关闭窗口。
7. 等待专注结束。
8. 查看统计记录。
9. 正常退出应用。

### 14.2 异常测试

- 专注期间关闭主窗口。
- 专注期间从托盘尝试退出。
- 专注期间重启应用。
- 专注期间电脑睡眠后唤醒。
- 白名单为空时开始专注。
- 数据库文件不存在时首次启动。

### 14.3 性能测试

- 前台窗口监控间隔不低于 1 秒。
- 长时间运行 CPU 占用应保持较低。
- 连续运行 2 小时无明显内存增长。

## 15. 开发注意事项

- 不要在第一阶段就实现所有功能。
- 不要把 Windows API 调用写进前端。
- 不要默认强制关闭用户应用。
- 不要把“防退出”做成系统级锁死。
- 不要使用需要管理员权限的方案作为 MVP 基础。
- 不要引入云同步、账号系统等非必要复杂度。
- 每阶段完成后先验证，再进入下一阶段。
